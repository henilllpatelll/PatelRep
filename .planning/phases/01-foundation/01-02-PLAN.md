  <behavior>
    - `request()` accepts an optional `isRetry = false` parameter (not exposed on the `api` export)
    - On HTTP 401 when `isRetry` is false: call `supabase.auth.refreshSession()`, if successful retry once with `isRetry = true`
    - On HTTP 401 when `isRetry` is true: throw `new Error('Session expired. Please log in again.')` — no further retry
    - On any other non-ok status: throw with `error.detail` or `HTTP {status}` as before
    - On 401 where `refreshSession` returns an error: throw the session-expired error (do NOT recurse)
    - The `api` object export is unchanged — callers see no difference
  </behavior>
  <action>
    Rewrite the `request` function in `apps/mobile/lib/api/client.ts` to add the `isRetry` parameter and 401 branch:

    ```typescript
    async function request<T>(
      method: string,
      path: string,
      body?: unknown,
      isRetry = false
    ): Promise<T> {
      const headers = await getAuthHeader();

      const response = await fetch(`${API_BASE}${path}`, {
        method,
        headers,
        body: body ? JSON.stringify(body) : undefined,
      });

      if (response.status === 401 && !isRetry) {
        const { data, error } = await supabase.auth.refreshSession();
        if (!error && data.session) {
          return request<T>(method, path, body, true);
        }
        throw new Error('Session expired. Please log in again.');
      }

      if (!response.ok) {
        const error = await response.json().catch(() => ({ detail: 'Unknown error' }));
        throw new Error(error.detail ?? `HTTP ${response.status}`);
      }

      return response.json() as Promise<T>;
    }
    ```

    The `api` export at the bottom of the file is unchanged.

    Note: `getAuthHeader()` reads the current session from `supabase.auth.getSession()`. After `refreshSession()`, the Supabase client updates its internal session, so the recursive call to `request(..., true)` will pick up the new token via `getAuthHeader()`. This is why no manual token threading is needed.
